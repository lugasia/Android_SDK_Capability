FlycommC2 — Android SDK Capability Surface (Kotlin)

A full-function catalog of what the Android platform exposes across cellular, Wi-Fi, roaming, GNSS, QoS, satellite (D2C / NTN) and V2X — and what it structurally does not — for building the FlycommC2 sensing SDK. Every function is tagged with its minimum privilege tier and since-API level.

Scope. Companion to the Cell Observation Sequence per RAT reference. That doc covers the per-RAT cell-observation pipeline in depth; this one is the horizontal API map — every domain the SDK can sense, the concrete Kotlin/Java entry points, the permission/privilege each needs, and the analytic use (anomaly detection, QoS, GNSS integrity) each unlocks.

Target floor. API 33 (Android 13) as the minimum; each function is annotated with its own since-API so you can raise or lower the floor per feature. Where a capability only appears in API 34/35/36/37 it is called out inline.

Headline finding. The four sensing domains split cleanly by how much the OS trusts an app. Wi-Fi, GNSS raw measurements and cellular identity/signal are richly exposed to an ordinary (location-permitted) app. Detailed 3GPP QoS (QCI / 5QI / QFI / GBR) is SYSTEM-only — the QosCallback API is @SystemApi; a normal app gets only coarse link-bandwidth estimates plus whatever it measures itself. Satellite/NTN is almost entirely opaque — a normal app gets only a "state may have changed" listener (API 36) and a boolean NTN flag (API 35); all real satellite control is SYSTEM-only. C-V2X (PC5 sidelink) is not exposed by any public Android API at all. Build the SDK around the green tiers; treat the red ones as requiring a privileged fleet build or external hardware.

How to read this doc

Every catalog row carries three tags: the exposure status (can a real app get this value?), the minimum privilege tier required, and the since-API level. The privilege tier is the decisive axis — it is what determines "exposed vs not" for your chosen fleet posture.

Exposure status

PUBLIC value available to an app on the stated tier  ·  PARTIAL available but degraded — throttled, coarse, vendor-dependent, or some fields null  ·  NONE not reachable from an app of the stated build — SystemApi, @hide, or absent  ·  N/A concept doesn't apply

Privilege tiers

NORMAL  Play-Store-installable app. Manifest + runtime permissions only (READ_PHONE_STATE, ACCESS_WIFI_STATE, …). No signing requirement.
LOCATION  As NORMAL, but additionally requires ACCESS_FINE_LOCATION (and location services on). Most identity/scan data sits here.
CARRIER  Carrier privileges granted by rules on the UICC (SIM). No platform signing needed, but you need cooperation of the SIM issuer. Unlocks extra telephony reads/controls.
SYSTEM  Preinstalled/privileged or platform-signed app holding signature-level permissions (e.g. MODIFY_PHONE_STATE, SATELLITE_COMMUNICATION). Feasible on a controlled/MDM fleet, not on the Play Store.
AOSP/HAL  Only reachable by modifying the platform itself — custom ROM, vendor RIL/HAL extension, or external radio hardware. Not an app-level path.

Reading rule: a row tagged SYSTEM or AOSP/HAL is effectively NONE for a Play-Store SDK, and becomes PUBLIC only if you ship a privileged build. That is the core "exposed vs not" decision.

1. The privilege-tier model in practice

Android gates telemetry on a ladder. The same physical fact (e.g. serving-cell identity) is available at different fidelities depending on where your SDK sits on that ladder. The four practical postures for a FlycommC2 deployment:

PostureTier reachedHowWhat it unlocks over the tier below
Consumer SDK (Play Store)NORMAL + LOCATIONRuntime permission grantsFull Wi-Fi scan, GNSS raw + AGC, CellInfo identity/signal, coarse link QoS, NTN boolean flag.
SIM-partnered SDK+ CARRIERCarrier privilege rules on the UICCNetwork-scan control, some service-state and configuration reads without platform signing.
Managed fleet / MDM build+ SYSTEMPreinstalled privileged or platform-signed APKFull SatelliteManager (datagrams, pointing, NTN signal strength, modem state), 3GPP QoS session attributes (QCI/5QI/GBR via QosCallback), privileged telephony, richer physical-channel data.
Custom device / RSU+ AOSP/HALCustom ROM or vendor HAL, or external modem/SDRForced band scans (Pattern-B unlicensed-band), PC5 sidelink / C-V2X, raw IQ, cellBarred, SIB19 NTN ephemeris.
Design implication. Ship one SDK core that targets NORMAL+LOCATION and degrades gracefully, then a privileged "extension module" that lights up the SYSTEM surface only when the host build grants it. The existing per-RAT doc's SDK-vs-RSU coverage split is exactly this ladder applied to cell scanning.

2.4 What cellular does NOT expose to any app

  • cellBarred, q-RxLevMin, intraFreqReselection, si-SchedulingInfo (SIB1) — NONE. The modem consumes them; no API surfaces them. BarringInfo is the closest and it is your-side, not broadcast.
  • Neighbor SIB lists (SIB4/5/11) — PARTIAL, only if the vendor RIL re-emits neighbors as extra CellInfo rows. Not guaranteed.
  • NR beam ID, BWP, SCS, S-NSSAI slice — NONE (S-NSSAI is NAS-level; beam/BWP are vendor-RIL only). See per-RAT doc §6.5.
  • Raw IQ / PHY samples, forced band scan outside SIM authorization — AOSP/HAL only. This is the Pattern-B blind spot from per-RAT §7.
  • additionalPlmns (MOCN, LTE/NR) — PUBLIC at API 30+ (CellIdentityLte/Nr.getAdditionalPlmns()) but currently dropped at ingestion; recommended DDL add.

4.3 What Wi-Fi does NOT expose

  • Device Wi-Fi MAC — PARTIAL. Randomized per-network; apps read 02:00:00:00:00:00. True MAC is SYSTEM.
  • Monitor mode / raw 802.11 frame capture — NONE. No packet-injection or promiscuous capture from an app; needs AOSP/HAL or external adapter.
  • Unthrottled continuous scanning — PARTIAL. Scan cadence is rate-limited by the framework; only a privileged app can bypass throttling.
  • Neighbor-AP CSI / per-subcarrier — NONE in public API.

QoS 6. Quality of Service

Two very different surfaces sit under "QoS", and the split is the whole story. Detailed 3GPP QoS session attributes (QCI, 5QI, QFI, guaranteed/maximum bit-rate) are delivered by the QosCallback API — which is @SystemApi and therefore SYSTEM-only. A normal app cannot register it (both android.net.QosCallback and NrQosSessionAttributes are absent from the public SDK). What a normal app can use is the coarse link view from NetworkCapabilities plus any latency/jitter/loss it measures itself.

6.3 What QoS does NOT expose

  • 3GPP QoS class (QCI / 5QI / QFI / GBR-MBR) on a consumer app — NONE. The QosCallback API that carries it is @SystemApi; only a privileged build can read it (§6.2).
  • QoS of other apps / the whole device — NONE even with the SystemApi. Callback scope is a single owned socket.
  • End-to-end latency / jitter / loss — NONE as an API on any tier. The SDK must measure it (RTT probes, timestamped echo).
  • Per-flow throughput class — PARTIAL; TrafficStats/NetworkStatsManager give byte counts (public), not the QoS class.

SAT 7. Satellite — Direct-to-Cell / NTN

This is the sharpest exposed-vs-not boundary in the whole surface. Android added a native satellite framework in Android 15 (API 35) and a small public SatelliteManager in Android 16 (API 36) — but almost the entire capability is SYSTEM. A normal app can detect that it is on a non-terrestrial network and be told the satellite state may have changed; it cannot provision, message, point, or read NTN signal strength.

What is invisible even to system apps

  • SIB19 NTN configuration (satellite ephemeris, common TA / propagation-delay compensation, epoch) — AOSP/HAL. Lives in the modem; not surfaced by SatelliteManager. Same class of gap as cellBarred.
  • Raw NTN doppler / per-beam measurements — NONE. Only the aggregate NTN signal strength (system) is offered.
Practical takeaway for FlycommC2. On a consumer SDK, treat satellite as a detection-only domain: log isNonTerrestrialNetwork() transitions and available-service contraction, and correlate with GNSS/cell context. Any active satellite sensing (signal quality, datagram, pointing, SIB19) requires a privileged fleet build or dedicated NTN hardware — mirror the SDK-vs-RSU split you already document for unlicensed-band.

V2X 8. V2X (C-V2X / PC5 sidelink)

There is no public Android API for C-V2X. Direct sidelink (PC5) — the BSM/CAM/DENM short-range vehicle messaging layer — is not exposed at the app level in AOSP. The Automotive stack (android.car) surfaces vehicle properties, not V2X radio access. This is a hard structural gap, not a permission gate.

11. SDK build recommendations

11.1 Two-layer SDK. Core targets NORMAL+LOCATION and delivers cellular, Wi-Fi, GNSS, roaming and coarse/self-measured QoS at full fidelity. A privileged extension module (compiled against the SystemApi stubs, active only when the host build grants it) lights up satellite control, the 3GPP QoS-attribute callback, forced band scans and RAT control. This mirrors the SDK-vs-RSU split already documented for unlicensed-band.

11.2 Prefer callback surfaces over polling. TelephonyCallback, registerScanResultsCallback, GNSS measurement callbacks and SignalStrengthUpdateRequest give event-driven, battery-cheap streams. Reserve requestCellInfoUpdate/getScanResults polling for forced refreshes.

11.3 Treat GNSS AGC + raw measurements as a first-class sensor. They are public (API 34 for AGC) and are the single best jamming/spoofing signal on the device. Probe getGnssCapabilities() at startup and record which fields the chipset supports so analysis can branch per-device.

11.4 Guard every API-level and privilege branch. Floor is API 33, but AGC is 35-effective-via-34, satellite public is 36, NTN flag is 35. Feature-detect (hasSystemFeature, capability probes, Build.VERSION.SDK_INT) rather than assuming; degrade to PARTIAL cleanly.

11.5 Externalize V2X. No handset path exists for PC5. If in scope, ingest from an external OBU/RSU over a socket; do not design for on-device sidelink.

11.6 Record the tier per sample. Persist which privilege tier produced each observation (as you persist appManipulated). Analysts then know whether a silent rule is a real negative or a coverage gap — the same "silent rule" problem flagged in the per-RAT doc.
